## Summary

### Responsibility

The Semester Settings area is the administrative control plane for all evaluation data in TEDU VERA. It manages the lifecycle of semesters (create, edit, delete, set-active), owns the per-semester evaluation criteria template and MÜDEK outcome mappings stored as JSONB in the `semesters` table, and controls the eval-lock toggle that gates juror scoring access. Every other area of the admin panel — Rankings, Grid, Analytics, Details, Juror Management, Project Management — derives its data from the currently selected semester. Changes to semester settings have cascading effects across the entire application.

### Most Critical Risks

1. **Eval-lock is not enforced on the backend for template updates.** `rpc_admin_update_semester` does not check `is_locked`. The only guard is in `useManageSemesters.handleUpdateCriteriaTemplate/MudekTemplate()`. A direct Supabase RPC call or a future code path that bypasses the hook can mutate templates on a locked/scored semester.
2. **Cascade delete is irreversible and immediate.** Deleting a semester removes all associated projects, scores, and juror auth records via `ON DELETE CASCADE` at the database level. There is no soft-delete, no recycle bin, and no per-field undo.
3. **Criteria template fallback is silent.** `getActiveCriteria([])` returns the global `CRITERIA` from `config.js` without any warning. A semester with an accidentally empty `criteria_template` silently scores against the wrong rubric.
4. **Stale-edit detection is UI-only.** The Realtime warning disables the save button, but there is no server-side optimistic-locking or last-write-wins timestamp enforcement on template updates.
5. **`loadSemesters()` retry is a fixed 600 ms sleep.** If the DB returns an empty list on first call, a single blind retry is made. If the second call also fails or returns empty, the UI silently shows no semesters.

### Overall Health Status

**Good with targeted risks.** The semester CRUD flow is well-structured, thoroughly validated at both the UI and DB layers, and has reasonable Realtime integration. The primary architectural concern is the soft nature of eval-lock enforcement on the backend for template RPCs. The cascade delete risk is mitigated by the delete-password requirement at the DB layer (separate from admin password) and the confirmation dialog in `useDeleteConfirm`, but no automated test covers that dialog flow. Test coverage is adequate for CRUD and sorting; significantly weaker for eval-lock, template save flows, and cross-area effects.

## Critical Findings

### CF-1: Eval-lock not enforced at the database layer

**RESOLVED 2026-03-23.** `rpc_admin_update_semester` in `sql/000_bootstrap.sql` now raises `semester_template_locked_by_scores` if the semester's `is_locked` flag is true when template fields are being updated. The DB-level guard is applied before the score-existence check, closing the bypass window for direct API calls and stale React state scenarios.

### CF-2: Silent criteria template fallback

**PARTIALLY MITIGATED 2026-03-23.** `ManageSemesterPanel` now displays a `.semester-default-template-badge` warning on semesters with an empty or missing `criteria_template`, so admins can identify which semesters use global defaults. The `getActiveCriteria` fallback itself is unchanged by design — scoring still works. A jury-side `console.warn` was added to `_loadSemester` to surface the fallback in dev tools. Full mitigation would require auto-seeding new semesters with `defaultCriteriaTemplate()` at creation time (deferred).

### CF-3: Realtime DELETE with open edit modal

**RESOLVED 2026-03-23.** The Realtime DELETE handler now calls `notifyExternalSemesterDelete`. `ManageSemesterPanel` auto-closes the edit modal when the deleted ID matches `editForm.id` and shows a warning banner. Covered by test `semester.realtime.01`.
